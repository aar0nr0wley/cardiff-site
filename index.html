Cardiff

A personal iOS health dashboard.

What it is

Cardiff is a single-user iOS app that consolidates personal health signals — lab results, vital signs, workouts, weight, and nutrition — into one timeline so trends are easier to see across categories.

It is built for personal use by the developer. There is no multi-user product, no marketplace, no advertising, and no third-party data sharing.

Health data Cardiff accesses

Read access (HealthKit):

  • Clinical Health Records — lab results (HKClinicalTypeIdentifierLabResultRecord) and vital signs (HKClinicalTypeIdentifierVitalSignRecord) imported into iOS Health from the user's healthcare provider (Sutter Health, via Epic FHIR R4). Cardiff queries these records, parses the FHIR Observation resources locally, and displays the results in a labs timeline within the app.
  • Workout heart rate — heart rate samples recorded by Apple Watch during a workout, used to render the in-session live HR overlay and post-session HR chart.

Write access (HealthKit):

  • Workouts — sessions logged in Cardiff (exercise, sets, duration, heart rate) are written back as HKWorkout samples so the iOS Health app reflects the user's training history.
  • Active energy burned — calories estimated from logged workouts.
  • Dietary energy consumed — calories from food log entries.
  • Body weight — weigh-ins recorded in the app.

Cardiff does not write to Clinical Health Records. HealthKit reserves clinical record writes for healthcare providers, and Cardiff's clinical access is read-only by design.

How data is stored

Data read from HealthKit is stored in a private cloud database owned by the user (Postgres with row-level security). Every table enforces an auth.uid() = user_id policy so that only the authenticated owner of the data can read or modify their own rows. There is no shared multi-tenant pool of user data and no cross-user access.

There is no telemetry, no event tracking, no advertising network, and no analytics SDK. Data is never sold, shared, or transmitted to any third party except in the explicitly user-initiated cases described in the next section.

Third-party services

Cardiff transmits data to two external services. Both are single-purpose, both are scoped to direct user actions, and neither is used for advertising or analytics.

  • Cloud database (Postgres + auth) — used to store the user's own data. The project is owned by the user.
  • Anthropic Claude API — used optionally when the user invokes an AI guidance feature in the app, to provide personalized summaries and suggestions across the user's training, nutrition, and (when the user opts in) lab and vital sign history. Data is sent only when the user explicitly triggers an AI feature — nothing is transmitted in the background. The user can disable AI features at any time, and can choose which categories of data each feature is allowed to include. Per Anthropic's API policy, prompts and responses sent via the API are not used to train Anthropic models.

Privacy summary

  • HealthKit data is read with explicit user consent; clinical records are read-only.
  • Data is stored only in the user's own private database with row-level security.
  • No advertising, analytics, telemetry, or third-party tracking.
  • Data is sent to the Claude API only when the user explicitly invokes an AI feature, and the user controls which categories can be included.
  • Data can be deleted by the user from within the app or directly via the database project.
